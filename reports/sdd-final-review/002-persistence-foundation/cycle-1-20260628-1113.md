**Additional medium findings (not in the critical/high table):**
- **A04 – Insecure Design / storage atomicity:** `append_event` allocates `seq` with `SELECT MAX(seq) ...` inside the INSERT and the comment at `src/gamebook/storage/postgres.py:230-232` claims "no race condition because we hold the row-level lock via the INSERT". In fact, the subquery does not lock the range, so concurrent writers (multiple processes/workers) can select the same `MAX(seq)` and one append will abort with a unique-constraint violation instead of being stored. The current single-threaded private event loop mitigates this for the stdio MCP server, but the design is not safe for multi-writer deployments.
- **A04:** `_build_snapshot` in `src/gamebook/storage/postgres.py:475-538` reads the mutable campaign state across multiple tables without an explicit transaction boundary, so the snapshot used by `save_slot` may be inconsistent under concurrent modification.

### Architecture Assessment

The implementation is architecturally sound at the macro level: `PostgresStorage` implements `StorageBackend` behind the interface, is injected only in `mcp.server.main()` (the composition root), and leaves the engine (`domain`, `rules`, `combat`, `mcp.tools`) untouched. The sync/async bridge via a dedicated daemon thread is the ADR-014 decision, and every mutating method wraps its work in `async with session.begin()` — honouring Principle V.

SOLID is mostly respected:
- **Single Responsibility**: persistence is isolated in `storage/postgres.py`; the schema migration is scoped to slice-002 engine tables only.
- **Open/Closed**: the new backend was added without changing the `StorageBackend` protocol or any engine consumer.
- **Liskov**: raw CRUD parity and restart-resume tests exist; `isinstance(pg, StorageBackend)` is asserted.
- **Dependency Inversion**: `combat` and `mcp` depend on `StorageBackend` (the latter via `TYPE_CHECKING`), not on `JSONStorage`/`PostgresStorage`.

The plugability audit was updated to include `gamebook.storage.postgres` and passes green (`15 passed` in `tests/qa/test_dependencies.py` + `tests/qa/test_isolation.py`). The full suite is green (`228 passed, 42 skipped` — skips are live-Postgres tests when `DATABASE_URL` is absent).

However, production hardening and edge-case parity are incomplete:
- **Resource lifecycle**: no `dispose()`/`close()` path for the async engine or daemon thread, so tests and long-running processes leak threads/connections.
- **Concurrency**: `append_event` seq allocation is not serializable despite the comment claiming it is.
- **Consistency**: `_build_snapshot` reads mutable state across multiple tables without an explicit transaction boundary.
- **Security**: PostgreSQL connections are not TLS-hardened by default.
- **Input parity**: string identifiers (`slot` names, `combat_id`) are not validated like `JSONStorage` does.

These are not architecture-breakers, but they prevent the slice from being production-ready.

### Cross-Report Synthesis

The QA and Security findings converge on the same conclusion: the slice is functionally complete and architecturally clean, but it is not production-hardened.

**Isolated issues:**
- Test coverage gaps (`test_storage_swap.py` omits Postgres; `test_atomic_writes.py` raises before any `execute()`).
- Missing input validation for empty/`None` slot/combat identifiers.
- Missing deterministic cleanup for engines/threads.

**Systemic issues (QA + Security overlap):**
- `append_event` seq generation is a concurrency bug under both QA and Security scrutiny; the misleading comment makes it worse.
- `_build_snapshot` reads across tables without an explicit transaction, creating inconsistent snapshot risk.
- No TLS enforcement is a Security HIGH finding that also affects production deployability.

No finding breaks the swap boundary or the interface contract. The architecture is correct; the defects are implementation-level hardening and parity gaps.
