## Tech Leader Verdict

### Architecture Assessment

The 003 web backend MVP demonstrates **sound architectural bones that are undermined by a fundamental multi-tenancy flaw and unimplemented remediation**. The dependency-direction discipline (the project's golden rule) is preserved: `src/gamebook_web` depends only on `MCPToolset` (the MCP protocol abstraction) and never imports concrete storage or engine internals — confirmed green by the plugability audit (59 passed). The `NarratorBackend` port with a deterministic `FakeNarrator` (ADR-011) is correctly realized, enabling the play loop to be tested without an LLM. The `Scene` → `effects[]` → MCP tool flow structurally enforces Principle I (numbers never in prose) at the harness boundary, and the output validator in `agent.py` is a code-level invariant independent of the model.

However, the architecture has a **critical scalability and isolation defect at its core**: the engine `MCPToolset` is started ONCE at app startup (`app.py:42-44`), scoped to a single `GAMEBOOK_CAMPAIGN_ID` read from server-side env at boot, and stored in `app.state.engine_toolset`. Every route retrieves that SAME shared toolset via `get_engine_toolset(request)` regardless of the `campaign_id` in the URL path. The URL `campaign_id` is NEVER forwarded to the engine — none of the ~20 `call_engine(...)` call sites in `play.py`/`combat.py` pass it, and `build_server` in `mcp/server.py:74` takes a single `storage` instance, not a `storage_factory`. This means the engine is **single-tenant by construction**: all campaigns share one character sheet, one world, one event log, one combat state. `_campaign_or_404` only guards the in-memory registry (ownership + existence) — it does NOT scope engine state. This is a cross-account data leak (OWASP A01) and makes multi-campaign support impossible. This is the root cause of multiple QA + Security findings.

Complexity is otherwise well-managed: the `mcp_host.py` factory seam allows test injection of an in-process toolset; the `CampaignRegistry` is a clean dev stub with a documented swap path to 004; the effect-to-tool mapping (`EFFECT_TO_MCP_TOOL`) keeps `Scene.effects[]` in lockstep with the MCP contract. The combat subagent delegation (ADR-001 pattern) is correctly structured but entirely untested.

**SOLID assessment**: SRP is mostly respected (route modules, harness, auth, sessions are separate). DIP is honored (interfaces only). OCP is served by the swap seams. The violation is LSP/ISP-adjacent: the `MCPToolset` abstraction is used as if it were per-campaign, but the concrete wiring is singleton — the contract lies about its multi-tenancy.

- ✅ **ADR-009 (Swap boundary tests through consumer)**: Compliant for storage; partially violated — no multi-campaign isolation test exists (a temporary `test_campaign_isolation_check.py` was written to prove the leak, then deleted; only its `.pyc` artifact remains in `__pycache__`).

### Cross-Report Synthesis

**Pattern 1 — The single-shared-toolset is the root cause of multiple findings.** The QA "HIGH: engine state not scoped per campaign" and the Security "CRITICAL: cross-account data leakage" + "CRITICAL: GAMEBOOK_CAMPAIGN_ID is server-side env, never derived from the request" are the same defect viewed through different lenses. **Pattern 2 — ADRs as documentation without implementation.** ADRs 017–020 are well-reasoned and correctly scoped, but they exist only as untracked files on `dev`. The 006-cycle1-remediation slice has 35 tasks across 6 phases — ALL unchecked, NONE merged. The code on `dev` is identical to what the prior cycle-1 review (20260628-0752) found. This is a process gap: decisions were recorded but the execution slice was never started. The remediation is planned, not delivered.

**Pattern 3 — Contract drift between backend and frontend.** Five HIGH findings (TurnResponse, CombatRoundResponse, FleeCombatResponse, campaign_id vs id, missing endpoints) all stem from the backend and frontend being developed in parallel (003 and 005) without a shared contract enforcement mechanism. ADR-017 ("backend wins") is the right call, but without a live integration test (FR-015/T014) the drift will recur. The frontend e2e test runs only in mock mode (`play-loop.spec.ts` explicitly `VITE_USE_MOCK=true`), so the drift was never caught.

**Pattern 4 — Production guards are absent across the board.** No boot-time assertion against `GAMEBOOK_DEV_MODE=1` in production, no `/docs` disable, no auth-failure logging, no rate-limiter keying on account_id. Each is individually MEDIUM, but together they form a pattern: the dev stubs were built without hard fail-closed guards for production misconfiguration. The 006 slice addresses this (FR-008/009/010) but, again, unimplemented.

**Pattern 5 — Dev auth is a stub with no guardrails.** The `dev-token` is static, public (shipped in the frontend bundle), has no expiry, and `GAMEBOOK_DEV_MODE=1` disables auth with no production assertion. Combined with the shared engine, anyone with `dev-token` gets all game state. The Security report correctly rates this CRITICAL. The 004 branch (unmerged to `dev`) adds OIDC but does NOT fix the shared-engine issue — so even merging 004 would not close the leak.

**Pattern 6 — Governance debt.** Duplicate ADR numbers (014 ×3, 015 ×2) and the dev-token mismatch across three files (`dev-token` / `dev-stub-token` / `dev-token-grimoire`) indicate that cross-file consistency checks are not being enforced. ADR-020 addresses the numbering; T025 addresses the token — both pending.

**Isolated bugs vs Systemic issues:**
- The contract mismatches are systemic — they stem from parallel development without integration testing.
- The single-toolset issue is systemic — it's an architectural constraint, not a bug.
- The missing endpoints are systemic — they represent incomplete implementation of FR-025 and FR-007.
- The combat victory path gap (MEDIUM) is isolated — a logic omission in one route.
- The `request: Request = None` default is isolated — a minor FastAPI pattern issue.
