### Numbers-Never-In-Prose Audit
**The defining rule is preserved in the web path and frontend, with one structural caveat.**

Evidence the rule holds:
- `Scene.effects[]` carry operation *parameters* only, never result values — enforced by schema docstring and `EFFECT_TO_MCP_TOOL` lockstep (`scene.py:1-12,23-33`)
- `PydanticNarrator.output_validator` (`agent.py:136-144`) raises `ModelRetry` if `_scene_contains_fabricated_numbers` detects result keys (`result`, `total`, `roll`, `damage`, etc.) in effect params or asserted stat values in narrative prose (`agent.py:70-91`)
- All numeric outcomes in API responses come from `call_engine()` MCP tool results (`play.py:116-149`, `combat.py:117-122`)
- `test_scene_numbers.py:196-260` proves character stats in turn responses equal engine-rolled stats (narrator can't change them without an effect)
- Frontend components are pure renderers: `CharacterSheet.tsx:1-8`, `CombatPanel.tsx:1-10`, `Inventory.tsx:1-8`, `MapPanel.tsx:1-8` — all values come from typed API props, nothing computed client-side
- `no-fabricated-values.test.ts` audits mock fixtures for engine-realistic ranges
- Mock data in `mock.ts` uses valid engine ranges (skill 7-12, stamina 14-24, combat AS = skill + 2d6)

Caveats:
- The `_RESULT_KEYS` denylist (`agent.py:63-67`) is heuristic, not exhaustive — a narrator could fabricate a number under an unrecognized key. This is a defense-in-depth gap, not a current violation.
- The `FakeNarrator` default scenes (`base.py:60-101`) contain no numbers in prose — compliant.
- The mock combat round (`mock.ts:335-342`) hardcodes `hero_attack: 15` etc., but these are documented as simulated engine output with valid ranges — acceptable for mock mode.

### Auth & Access Control Assessment

**Dev auth stub gating (`dev_auth.py`):**
- ✅ Fail-closed by default: `GAMEBOOK_DEV_MODE` defaults to `"0"`; missing token → 401 when dev mode is off.
- ✅ The seam design (`Depends(get_current_account)`) is clean — slice 004 can swap the implementation.
- ❌ **No hard guard preventing production deployment with dev auth.** There is no check like `if os.getenv("GAMEBOOK_DEV_MODE") and os.getenv("ENV") == "production": raise RuntimeError(...)`. The only protection is operational discipline. A misconfigured deployment silently runs with a static token.
- ❌ **The static token `dev-token` is effectively public** — it appears in source, in `.env.local.example`, and the frontend ships a variant in the JS bundle.

**Campaign ID trust (`GAMEBOOK_CAMPAIGN_ID`):**
- ❌ **CRITICAL: The campaign ID is a server-side env var, not derived from the request.** The engine subprocess is scoped to one campaign at boot. The per-request `campaign_id` path parameter is never passed to the engine. This means:
  1. The engine always operates on the boot-time campaign regardless of which `campaign_id` the client requests.
  2. The ownership check in `_campaign_or_404` (`play.py:84`) checks the in-memory registry, but the actual engine state (character, world, events) is shared across all campaigns.
  3. **A client cannot override `GAMEBOOK_CAMPAIGN_ID`** (good — it's server-side env, not a header), but the design means the engine cannot serve multiple campaigns anyway. The multi-campaign API is a facade over a single-campaign engine.
- ❌ **`GAMEBOOK_CAMPAIGN_ID` is not validated as a UUID** — it's passed raw to `PostgresStorage` and used in parameterized queries (no SQL injection, good), but no format validation.

**IDOR risks:**
- The `_campaign_or_404` ownership check (`play.py:77-89`) is correct *in principle* — it verifies `state.account_id != account.account_id` and returns 403. However, since the registry is in-memory and the engine is shared, the check is theater: the engine data returned after the check belongs to the boot campaign, not the requested one.

**FR-025 enforcement:**
- ❌ **Not implemented in the backend.** The frontend has full lease-handling UX (`useGame.ts:73-87`, `SessionConflict.tsx`), but the backend has no `/session`, `/session/takeover`, or `/session` DELETE endpoints. The `CampaignRegistry` has no lease field. These frontend calls will 404 against the real backend.

### CORS & Misconfiguration
- ✅ **CORS is restrictive by default** — `GAMEBOOK_CORS_ORIGINS` defaults to empty string, which disables CORS middleware entirely (`app.py:123-128`). No wildcard origin.
- ✅ Origins are explicitly configured via comma-separated env var.
- ⚠️ `allow_methods=["*"]` and `allow_headers=["*"]` with `allow_credentials=True` — broader than necessary but not a vulnerability given the explicit origin list.
- ❌ **No production guard on `/docs`, `/redoc`, `/openapi.json`** — always enabled (`app.py:96-98`). Should be gated on a non-production env var.
- ❌ **No `debug` mode flag check** — the app doesn't set `debug=True` explicitly (good), but the quickstart recommends `--reload` which is a dev-only flag.
- ✅ **Generic exception handler does not leak internals** — returns `"An internal server error occurred."` (`app.py:173`). Good.
- ✅ **Narrator errors are sanitized** — `play.py:343-346` returns a generic `invalid_scene` message, not the raw exception. Comment explicitly notes "Do not leak internal exception details." Good.
- ⚠️ **`create_character` error mapping inspects raw engine message text** (`play.py:256`: `if "living character" in msg`) — string matching on exception messages is fragile but not a security risk.

### Schema Validation (FR-014)
- ✅ **Scene is Pydantic-validated.** `harness/scene.py` defines `Scene`, `Choice`, `Effect` as Pydantic models with a `field_validator` enforcing non-empty narrative (`scene.py:90-95`).
- ✅ **Effect types are a closed enum** (`EffectType` Literal, `scene.py:37-47`), bound to `EFFECT_TO_MCP_TOOL` mapping. An unknown effect type is rejected by Pydantic before reaching the engine.
- ✅ **Output validator rejects fabricated numbers** (`agent.py:70-91`) — scans `Effect.params` for result keys (`result`, `total`, `roll`, `damage`, etc.) and rejects prose asserting stat values via regex. Raises `ModelRetry` to force the model to retry.
- ⚠️ **The fabricated-number heuristic is bypassable.** The `_RESULT_KEYS` set is a denylist, not an allowlist. A narrator could include numbers under keys not in the set (e.g. `"hp"`, `"value"`, `"amount"`). The regex for prose (`agent.py:84`) only catches `stamina|skill|luck|gold` followed by `is|was|becomes|dropped to` — it misses "you lost 5 points", "took 3 damage" in prose, and any number in a choice label. This is a defense-in-depth gap, not a complete failure.
- ⚠️ **Effect `params` is `dict[str, Any]`** (`scene.py:70`) — untyped. The engine MCP tools receive these params and must validate them. The closed effect-type enum limits the blast radius, but a malicious narrator could send unexpected param shapes (e.g. `start_combat` with a string `enemy_skill` instead of int). The MCP tool layer would need to validate — this was not inspected in `server.py` tool signatures (which use typed Python params, so FastMCP should coerce/reject).
- ✅ **Scenes are never persisted directly** — only `scene_dict` is stored in the in-memory registry for `GET /scene` re-fetch (`play.py:375`). Effects are applied through MCP tools, and engine state is the source of truth. Invalid scenes raise 422 before any effect is applied (`play.py:348-353`).

### SSRF Assessment (A10)
- ✅ **No unvalidated URL fetching.** The narrator (`agent.py`) calls MCP tools and the Anthropic API — no arbitrary URL fetching from user input. The `StdioTransport` in `mcp_host.py` runs a local subprocess (`sys.executable -m gamebook.mcp.server`), not a remote URL. No SSRF surface found.

### SQL Injection Assessment (A03)
- ✅ **All SQL uses parameterized queries.** `postgres.py` exclusively uses `text("... :param ...")` with `{"param": value}` dicts. No string concatenation, no f-strings in SQL, no `.format()`. Grep for injection patterns found zero matches. This is correctly implemented across all 30+ query sites.

### SQLAlchemy Session Lifecycle
- ✅ **Sessions use context managers** (`async with self._session() as session: async with session.begin():`) — proper cleanup, no leaks.
- ✅ **`expire_on_commit=False`** (`postgres.py:99`) — avoids lazy-load-after-commit errors.
- ⚠️ **The sync/async bridge runs a private daemon event loop** (`postgres.py:78-84`) — `future.result()` blocks the calling thread. If the daemon loop dies, all subsequent calls hang indefinitely (no timeout on `future.result()`). This is a reliability/DoS concern, not a security vulnerability per se, but a stuck loop would hang the entire engine.
- ⚠️ **`_load_combat` does not use `FOR UPDATE`** (`postgres.py:307-311`) while `_save_combat` does (`postgres.py:332`). A read between another transaction's read and write could return stale combat state. Minor concurrency issue.

## Tech Leader Verdict

### Architecture Assessment

**Overall Assessment: The architecture is fundamentally sound in principle but critically broken in execution.**

The system follows a clean modular design with well-defined swap boundaries (storage, adventure module, harness) and strong separation of concerns. The MCP tool contract is stable, the domain layer is Pydantic-validated, and the "numbers never in prose" principle is encoded via output validators. However, the web backend implementation contains **architectural violations that break the multi-campaign isolation guarantee** — a fundamental requirement for a production system.

**SOLID Principles:**
- **Single Responsibility**: Well-maintained at the engine layer. Each module has a clear purpose (rules, storage, combat, mcp, harness).
- **Open/Closed**: Swap boundaries #1 (StorageBackend) and #3 (NarratorBackend) are properly abstracted via Protocols/ports. The PydanticAI narrator behind NarratorBackend is a good example.
- **Liskov Substitution**: The storage swap boundary is proven via ADR-009's three-backend test (InMemory, JSON, Mock). PostgresStorage correctly implements the interface.
- **Interface Segregation**: Clean interfaces — StorageBackend, RandomSource, CombatEngine, NarratorBackend. No fat interfaces.
- **Dependency Inversion**: The engine depends only on interfaces. The web layer correctly depends on MCPToolset abstraction.

**Scalability Concerns:**
- **CRITICAL**: Single shared engine toolset across all campaigns (`app.py:42-44`, `mcp_host.py:45-56`). This is a fundamental scalability blocker — the architecture cannot support multiple concurrent campaigns because the engine is scoped to a single `GAMEBOOK_CAMPAIGN_ID` at process startup.
- **In-memory CampaignRegistry**: Not durable, not production-ready (acknowledged as dev stub, but no guard prevents production deployment).
- **No session lease enforcement**: FR-025 is not implemented in the backend despite frontend having full UX for it.

**Complexity:**
- The sync/async bridge in PostgresStorage (ADR-014) is a necessary complexity given the constraints, well-documented and correctly implemented.
- The mock mode fixture layer (ADR-016) adds test complexity but provides good value for isolated testing.
- The output_validator denylist in agent.py is a heuristic complexity that should be replaced with an allowlist or stronger validation.
