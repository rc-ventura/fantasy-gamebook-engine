# ADR-009: Prove swap boundary #1 through the consumer, across three backends incl. an independent mock

**Status**: Accepted
**Date**: 2026-06-21
**Related spec**: [03-storage](../03-storage.md), [04-combate](../04-combate.md), [CONTRACTS §1/§4/§5](../CONTRACTS.md)
**Code**: `tests/qa/test_storage_swap.py`

---

## Context

Swap boundary #1 is `StorageBackend`: `JSONStorage` (Phase 1) must be replaceable
by `PostgresStorage` (Phase 2) with no other module changing. The infra team
already proves `InMemoryStorage` and `JSONStorage` are observationally identical
at the *raw CRUD* level (`tests/server/test_storage_parity.py`). That is
necessary but not sufficient for QA: the boundary that actually matters is whether
the engine's real *consumer* — `CombatService` — behaves identically no matter
which backend persists state. A boundary can be intact at the save/load level yet
broken if the consumer happens to rely on an incidental behaviour of one concrete
impl (e.g. returning a live reference instead of a snapshot).

## Decision

Test the boundary **one layer up, through the consumer**, and **parametrize across
three backends** — two real (`InMemoryStorage`, `JSONStorage`) and one written
purely for the test (`MockStorage`, a dict-backed `StorageBackend`):

- Drive a **single, scripted, deterministic fight** (`CombatService` with a fresh
  `random.Random(SEED)`) end to end against each backend.
- Capture the **full observable result** — every round outcome, the `FinalResult`,
  the final sheet, and that the in-progress combat was removed — and assert all
  three are byte-for-byte equal.
- **Exclude only the random `combat_id`** from comparison (it is generated by
  `uuid4`, not the injected RNG, so it differs per run by design).

```python
memory = _run_full_fight(InMemoryStorage())
json_  = _run_full_fight(JSONStorage(tmp_path / "estado"))
mock   = _run_full_fight(MockStorage())
assert memory == json_ == mock          # diverge => boundary violated
assert isinstance(MockStorage(), StorageBackend)   # structural conformance
```

The third, independent `MockStorage` is the key: if a brand-new implementation
written with no knowledge of the engine produces identical results, the engine
genuinely depends on the *interface* and nothing more. `MockStorage` deep-copies
on save/load (like `InMemoryStorage`) because snapshot-on-store is part of the
contract every faithful backend must honour.

## Alternatives considered

### Alternative A: rely on the existing raw save/load parity test only

**Why not chosen**:
- It proves the backends agree on CRUD, not that the *consumer* is decoupled. A
  consumer bug that leans on one impl's incidental behaviour would slip through.

### Alternative B: two backends only (memory + json)

**Why not chosen**:
- Both ship in this repo and could share an accidental assumption. A third,
  independent impl is a much stronger decoupling signal and a template for how
  `PostgresStorage` will be validated later.

### Alternative C: compare backends including `combat_id`

**Why not chosen**:
- `combat_id` is deliberately random per run; including it would force the test to
  fake/seed identity and obscure the real signal (game behaviour).

## Consequences

### Accepted

- Boundary #1 is proven at the level that matters (the consumer), and the test
  doubles as the validation recipe for any future backend (incl. `PostgresStorage`).
- A regression that couples `combate` to a concrete backend fails loudly with a
  three-way diff.

### Trade-offs

- Maintaining a third `MockStorage` is extra code, kept minimal and local to the
  test.
- The scenario is a fixed script; exotic combat paths still rely on the engine's
  own unit tests.

### Conditions that invalidate this decision

This decision should be **revisited** if:

1. `StorageBackend` grows stateful semantics that a trivial dict mock cannot
   faithfully emulate.
2. A property-based/contract-test harness for `StorageBackend` is introduced that
   subsumes the hand-rolled scenario.

### Migration path when needed

Promote `MockStorage` and `_run_full_fight` into a shared `StorageBackend`
contract-test suite parametrized over every impl (including `PostgresStorage`),
and have each backend register itself.
